## Overview

The string art generator uses a greedy algorithm to iteratively build a pin sequence. At each step, it picks the next pin that maximally reduces the difference between the target image and the current string art rendering. Starting from an initial pin, the algorithm repeats this process for up to `MAX_LINES` iterations, producing a sequence of pin indices that a physical thread would follow around the hoop.

<Note>
  The algorithm is greedy — it makes the locally optimal choice at each step without backtracking. This does not guarantee a globally optimal result, but it produces visually convincing string art in practice.
</Note>

## Error map

The algorithm works by maintaining an **error matrix** that tracks which parts of the image still need more string coverage.

For each pixel at position `(x, y)`:

```
error[x, y] = 255 - grayscale[x, y]
```

High values indicate dark areas of the target image that have not yet been covered by string. As lines are drawn, the error along those pixels is reduced. The algorithm prioritizes drawing lines through regions with the highest remaining error.

## Pin placement

`N_PINS` pins are evenly distributed around a circle. Each pin `i` is placed at angle `2π × i / N_PINS`, with a radius of `IMG_SIZE / 2 - 0.5` pixels:

```javascript
let center = length / 2;       // 250
let radius = length / 2 - 1/2; // 249.5
for (let i = 0; i < N_PINS; i++) {
    angle = 2 * Math.PI * i / N_PINS;
    pin_coords.push([
        Math.floor(center + radius * Math.cos(angle)),
        Math.floor(center + radius * Math.sin(angle))
    ]);
}
```

The half-pixel inset (`- 0.5`) keeps pin coordinates from landing on the very edge of the image, avoiding out-of-bounds access during line sampling.

## Line precomputation

Before the main loop runs, every valid pair of pins has its pixel coordinates pre-computed and cached. Pairs that are within `MIN_DISTANCE` of each other (by index) are skipped, since very short lines connecting adjacent pins add little visual value.

For each valid pair `(a, b)`, the pixel coordinates along the line are computed using linear interpolation (`linspace`), and stored in flat lookup arrays:

```javascript
for (b = a + MIN_DISTANCE; b < N_PINS; b++) {
    x0 = pin_coords[a][0]; y0 = pin_coords[a][1];
    x1 = pin_coords[b][0]; y1 = pin_coords[b][1];
    d = Math.floor(Math.sqrt((x1-x0)*(x1-x0) + (y1-y0)*(y1-y0)));
    xs = linspace(x0, x1, d);
    ys = linspace(y0, y1, d);
    line_cache_y[b*N_PINS + a] = ys;
    line_cache_y[a*N_PINS + b] = ys;
    line_cache_x[b*N_PINS + a] = xs;
    line_cache_x[a*N_PINS + b] = xs;
    line_cache_length[b*N_PINS + a] = d;
}
```

The cache is indexed by `a * N_PINS + b` (and its mirror `b * N_PINS + a`), so looking up the pixel path between any two pins is an O(1) array access during the main loop.

## Greedy selection

With the error map and line cache ready, the algorithm runs for `MAX_LINES` iterations. Each iteration selects the best next pin:

```javascript
for (l = 0; l < MAX_LINES; l++) {
    max_err = -1;
    best_pin = -1;
    for (offset = MIN_DISTANCE; offset < N_PINS - MIN_DISTANCE; offset++) {
        test_pin = (pin + offset) % N_PINS;
        if (last_pins.includes(test_pin)) continue; // avoid recent repeats
        line_err = getLineErr(error, ys, xs) * line_cache_weight[...];
        if (line_err > max_err) { max_err = line_err; best_pin = test_pin; }
    }
    line_sequence.push(best_pin);
    // Subtract LINE_WEIGHT from error along chosen line pixels
    error = subtractArrays(error, line_mask_with_LINE_WEIGHT);
    pin = best_pin;
}
```

The `getLineErr` function sums the error values at every pixel along a candidate line:

```javascript
function getLineErr(arr, coords1, coords2) {
    let result = new Uint8Array(coords1.length);
    for (i = 0; i < coords1.length; i++) {
        result[i] = arr.get(coords1[i], coords2[i]);
    }
    return getSum(result); // sum of error values along the line pixels
}
```

After the best pin is chosen, `LINE_WEIGHT` is subtracted from the error map along every pixel of that line (clamped to `[0, 255]`):

```javascript
function subtractArrays(arr1, arr2) {
    for (i = 0; i < arr1.selection.data.length; i++) {
        arr1.selection.data[i] = arr1.selection.data[i] - arr2.selection.data[i];
        if (arr1.selection.data[i] < 0) arr1.selection.data[i] = 0;
        else if (arr1.selection.data[i] > 255) arr1.selection.data[i] = 255;
    }
    return arr1;
}
```

This reduction means future iterations are less likely to draw lines through the same region, gradually filling in the dark areas of the image.

## Anti-repeat mechanism

The `last_pins` array tracks the 20 most recently visited pins. Any pin present in this list is skipped during candidate evaluation:

```javascript
last_pins.push(best_pin);
if (last_pins.length > 20) last_pins.shift();
```

Without this guard, the algorithm would tend to oscillate between the same two or three pins repeatedly, creating dense clusters of thread in one spot while neglecting the rest of the image.

## Output rendering

Selected lines are drawn on an internal `10000×10000` pixel OpenCV Mat (`IMG_SIZE * SCALE = 500 * 20`). Each line is rendered with a 2-pixel antialiased stroke:

```javascript
cv.line(result, p, p2, new cv.Scalar(0,0,0), 2, cv.LINE_AA, 0);
```

The high-resolution mat is then downscaled to `1000×1000` (`IMG_SIZE * 2`) for display on the canvas. Rendering at 10000px before downscaling produces smoother, higher-quality lines than drawing at display resolution directly.

The final pin sequence is written as a comma-separated list of integers:

```javascript
pinsOutput.value = line_sequence;
```

<Info>
  The Go implementation in `main.go` uses the same greedy algorithm with the same pin placement, line cache, and error-subtraction logic. It serves as a command-line reference implementation and outputs the pin sequence to stdout.
</Info>
